# Отчёт по Module 2: Dockerfile

**Цель модуля:** научиться упаковывать простое приложение в собственный Docker-образ с помощью Dockerfile.

**Результат:** создана папка `module2-dockerfiles` с FastAPI-приложением, Dockerfile и зависимостями. Образ собирается и запускается командами ниже.

---

## 1. Подготовка

Создана папка `module2-dockerfiles` со следующими файлами:

| Файл | Назначение |
|------|------------|
| `app.py` | простое FastAPI-приложение |
| `requirements.txt` | зависимости (`fastapi`, `uvicorn`) |
| `Dockerfile` | рецепт сборки образа |

### Приложение (`app.py`)

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/")
def root():
    return {"message": "Hello, World!"}


@app.get("/hello")
def hello(name: str = "World"):
    return {"message": f"Hello, {name}!"}
```

Эндпоинты:
- `GET /` — приветствие
- `GET /hello?name=...` — персонализированное приветствие

## 2. Теория: что такое Dockerfile

**Dockerfile** — текстовый «рецепт» сборки Docker-образа.  
В нём по шагам описывается: из какого базового образа взять окружение, какие файлы скопировать, какие команды выполнить при сборке и как запускать приложение при старте контейнера.

Команда `docker build` читает Dockerfile и создаёт образ — неизменяемый шаблон, из которого потом запускаются контейнеры.

---

## 3. Основные инструкции Dockerfile

Изучены и применены 6 основных инструкций.

### FROM

Задаёт **базовый образ**, от которого начинается сборка.

В проекте:
```dockerfile
FROM python:3.10-slim
```

Взял официальный Python 3.10 в облегчённой сборке (`slim`) — меньше размер образа, чем у полного `python:3.10`.

### WORKDIR

Устанавливает **рабочую директорию** внутри контейнера. Все последующие команды (`COPY`, `RUN`, `CMD`) выполняются относительно неё.

```dockerfile
WORKDIR /app
```

### COPY

**Копирует файлы** с хоста в файловую систему образа.

```dockerfile
COPY . .
```

Копирует всё из текущей папки хоста (рядом с Dockerfile) в `/app` внутри образа.

### RUN

Выполняет команду **во время сборки** образа. Результат сохраняется в слое образа.

```dockerfile
RUN pip install -r requirements.txt
```

Устанавливает зависимости FastAPI-приложения, чтобы их не нужно было ставить при каждом запуске контейнера.

### EXPOSE

**Документирует порт**, который слушает приложение. Сам по себе порт не открывает наружу — это делает `-p` при `docker run`.

```dockerfile
EXPOSE 5000
```

### CMD

Указывает команду, которая выполнится **при старте контейнера** (если не переопределена в `docker run`).

Использована **exec-форма** (JSON-массив) — best practice: процесс получает сигналы напрямую, без лишнего shell.

```dockerfile
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "5000"]
```

Почему не `CMD ["python", "app.py"]`: FastAPI — это ASGI-приложение, для его запуска нужен сервер `uvicorn`.  
`--host 0.0.0.0` нужен, чтобы приложение было доступно с хоста через проброс портов.

## 4. Итоговый Dockerfile

```dockerfile
FROM python:3.10-slim

WORKDIR /app

COPY . .
RUN pip install -r requirements.txt

EXPOSE 5000

CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "5000"]
```

---

## 5. Сборка образа

```bash
cd module2-dockerfiles
docker build -t my-app:1.0 .
```

| Флаг / аргумент | Значение |
|-----------------|----------|
| `-t my-app:1.0` | тег образа: имя `my-app`, версия `1.0` |
| `.` | контекст сборки — текущая папка (здесь лежит Dockerfile) |

**Критерий:** образ появляется в списке:
```bash
docker images
```

---

## 6. Запуск контейнера

```bash
docker run -d -p 8080:5000 my-app:1.0
```

| Флаг | Значение |
|------|----------|
| `-d` | запуск в фоне (detached) |
| `-p 8080:5000` | проброс порта `хост:контейнер` — на хосте открыт `8080`, внутри контейнера приложение слушает `5000` |
| `my-app:1.0` | образ для запуска |

**Критерий:** приложение доступно в браузере:

- http://localhost:8080 → `{"message":"Hello, World!"}`
- http://localhost:8080/hello?name=Docker → `{"message":"Hello, Docker!"}`
- http://localhost:8080/docs → Swagger UI FastAPI
