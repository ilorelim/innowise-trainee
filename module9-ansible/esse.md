# Эссе: Почему Ansible, а не Bash-скрипты?

**Задание:** объяснить, почему для конфигурации серверов удобнее Ansible, чем набор Bash-скриптов. Критерии: выделить **идемпотентность** и **декларативность**.

---

## В чём проблема Bash

Bash отлично подходит для разовых команд и короткой автоматизации. Но когда задача звучит как «на всех серверах должен быть nginx, нужный конфиг и сервис в `enabled`», скрипт быстро превращается в набор императивных шагов:

```bash
apt-get install -y nginx
cp nginx.conf /etc/nginx/nginx.conf
systemctl enable nginx
systemctl start nginx
```

Такой скрипт описывает **как** действовать, а не **какое состояние** нужно получить. Повторный запуск часто опасен или шумный: пакет уже установлен, файл уже на месте, сервис уже запущен — и скрипт либо падает, либо делает лишнюю работу, либо тихо ломает конфиг поверх уже изменённого.

Чем больше хостов и чем дольше живёт инфраструктура, тем дороже поддерживать «ручные» скрипты: ветвления `if`, проверки существования файлов, разный порядок команд на разных дистрибутивах.

## Декларативность: «я хочу nginx»

Ansible — **декларативный** инструмент. В playbook вы описываете желаемое состояние системы, а не пошаговый рецепт «сделай A, потом B»:

```yaml
- name: Ensure nginx is installed and running
  hosts: web
  become: true
  tasks:
    - name: Install nginx
      ansible.builtin.apt:
        name: nginx
        state: present

    - name: Deploy nginx config
      ansible.builtin.copy:
        src: nginx.conf
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: "0644"

    - name: Ensure nginx is enabled and started
      ansible.builtin.service:
        name: nginx
        enabled: true
        state: started
```

Смысл записи: «nginx должен быть установлен», «конфиг должен совпадать с эталоном», «сервис должен быть включён и запущен». Модули Ansible сами решают, какие команды выполнить на конкретной ОС. Вам не нужно вручную писать `apt-get` vs `yum`, проверять коды возврата и собирать логику «уже есть / ещё нет».

Bash говорит: *сделай эти команды*. Ansible говорит: *приведи систему к этому состоянию*.

## Идемпотентность: повторный запуск не ломает

**Идемпотентность** — свойство операции: повторное применение даёт тот же результат, что и первое, без побочных эффектов «накопительного вреда».

| Подход | Первый запуск | Второй запуск |
|--------|---------------|---------------|
| Bash «в лоб» | ставит nginx, копирует конфиг, стартует сервис | может снова `apt-get`, перезаписать живой конфиг, лишний раз `restart`, упасть на «уже установлено» |
| Ansible | приводит систему к нужному состоянию | видит, что всё уже так → `ok` / `changed=0`, ничего не ломает |

Модули Ansible спроектированы так, чтобы сначала проверить текущее состояние и менять систему только при расхождении. Поэтому playbook можно смело гонять много раз: после деплоя, после отката, в cron, из CI — конфигурация сходится к одному и тому же виду.

В Bash идемпотентность тоже можно «накодить» (`dpkg -l | grep`, `diff` перед `cp`, `systemctl is-active`), но это ручная работа на каждый шаг. Один забытый `if` — и скрипт перестаёт быть безопасным при повторе. В Ansible идемпотентность — норма модулей, а не отдельный слой проверок в каждом скрипте.

## Agentless и практический бонус

Ansible **agentless**: на управляемых хостах не нужен свой демон. Управление идёт по SSH (или `local` для localhost). Достаточно Python на целевой машине и доступа по ключу. Это проще, чем разворачивать агентов на каждом сервере, и ближе к привычной модели «зашёл по SSH и настроил», только воспроизводимо и масштабируемо через inventory.

## Вывод

Bash хорош для локальных утилит и одноразовых действий. Для конфигурации инфраструктуры Ansible выигрывает за счёт двух свойств:

1. **Декларативность** — описываете желаемое («хочу nginx»), а не цепочку команд.
2. **Идемпотентность** — повторный прогон безопасен и приводит систему к тому же состоянию, не ломая уже настроенное.

Вместе это даёт воспроизводимую, повторяемую и понятную автоматизацию вместо хрупких скриптов «запусти один раз и молись».
