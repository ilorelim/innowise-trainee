# Эссе: Почему Docker Compose — это не Production

**Задание:** объяснить, почему Docker Compose удобен для разработки и демо, но недостаточен для production. Критерии: выделить **Self-healing**, **Scaling** и **Zero-downtime deployment**.

**Контекст (М6 З10):** `docker compose up --scale app=3` — как балансировать трафик? Что будет, если один контейнер упадёт?

---

## Чем Compose хорош — и где заканчивается его зона

Docker Compose отлично описывает локальный стек: `app` + `db` + `redis`, сети, volumes, healthcheck, `.env`. Одна команда `docker compose up` поднимает всё окружение. Для обучения, CI-smoke и разработки этого достаточно.

Production требует другого: сервис должен **сам восстанавливаться**, **масштабироваться без ручной магии** и **обновляться без простоя**. Compose эти задачи либо не решает, либо решает частично и хрупко. Ниже — три ключевые дыры.

## 1. Self-healing: Compose не «чинит» упавший контейнер

**Self-healing** — способность платформы заметить отказ инстанса и автоматически вернуть желаемое число рабочих реплик.

В Compose можно задать `restart: unless-stopped` или `always`: если процесс внутри контейнера завершился, Docker Engine попробует перезапустить **этот же** контейнер. Но это не полноценное самовосстановление оркестратора:

| Ситуация | Compose | Оркестратор (Swarm / Kubernetes) |
|----------|---------|----------------------------------|
| Процесс в контейнере упал | `restart`-политика может поднять контейнер снова | контроллер поддерживает `replicas` |
| Контейнер удалён / нода умерла | Compose сам новые реплики не создаёт | создаётся новый Pod / task |
| Инстанс «жив», но не отвечает (hung) | без отдельного health + restart_policy эффект слабый | readiness/liveness → исключение из трафика и пересоздание |

Контекст из задания: при `--scale app=3` три контейнера работают «рядом». Если один упал, Compose **не гарантирует**, что снова станет три здоровых инстанса и трафик на мёртвый уйдёт только из балансировщика. Часто остаётся два живых + один мёртвый (или простаивающий), пока человек не вмешается (`docker compose up`, `ps`, логи).

В production «упал — и ждём админа» неприемлемо. Нужен контроллер желаемого состояния: «должно быть N здоровых реплик» — и платформа сама к этому сходится.

## 2. Scaling: реплики есть, балансировки «из коробки» нет

Команда масштабирования выглядит просто:

```bash
docker compose up -d --scale app=3
```

В `docker compose ps` действительно появятся три инстанса `app`. Но дальше всплывают ограничения:

1. **Конфликт портов.** Если в `docker-compose.yml` задано `ports: "5000:5000"`, на хосте порт `5000` может занять только **один** контейнер. Остальные реплики либо не стартуют с publish, либо требуют другой схемы публикации.
2. **Нет встроенного load balancer.** Compose создаёт DNS-имя сервиса (`app`), но при нескольких контейнерах клиенту всё равно нужна точка входа, которая размазывает запросы. Без reverse-proxy (Nginx, Traefik, HAProxy) или без режима Swarm с routing mesh «просто `--scale`» не даёт нормального распределения трафика с хоста.
3. **Масштаб = один хост.** Compose масштабирует контейнеры на **той же машине**, где запущен Compose. Нет распределения по кластеру нод, нет учёта ресурсов ноды, нет антиаффинити («не клади все реплики на один диск/один сервер»).

Итог: `--scale app=3` — удобный учебный жест, а не production-scaling. В бою нужны сервис discovery, балансировщик, квоты ресурсов и (часто) несколько машин. Это зона Kubernetes / Swarm / Nomad, а не одиночного `docker-compose.yml`.

## 3. Zero-downtime deployment: обновление = простой или ручной цирк

**Zero-downtime deployment** — выкатить новую версию так, чтобы пользователи не ловили 502 и обрыв сессий: старые инстансы ещё обслуживают трафик, новые поднимаются, health проходит, трафик переключается, старые гасятся.

Типичный путь в Compose:

```bash
docker compose up -d --build
```

Compose пересоздаёт контейнеры сервиса. На практике это часто выглядит так: старый контейнер останавливается → поднимается новый. В окне между stop и ready приложение **недоступно**. Нет штатного rolling update: «по одной реплике, с ожиданием healthy, с откатом при провале».

Можно вручную городить схемы (два compose-проекта, внешний Nginx, blue/green скриптами), но это уже самодельный оркестратор. В Kubernetes есть `RollingUpdate`, readinessProbe, surge/unavailable; в Swarm — `update_config` / `rollback_config`. Compose такого контракта не даёт «из коробки».

Без zero-downtime каждый релиз — риск минуты простоя и ночных деплоев «когда никто не смотрит». Для production это слабое место.

## Сводная таблица

| Требование production | Docker Compose | Что нужно вместо / сверху |
|-----------------------|----------------|---------------------------|
| **Self-healing** | Ограниченный `restart`; нет поддержания N реплик при падении | контроллер реплик, health → recreate |
| **Scaling** | `--scale` на одном хосте; конфликт `ports`; нет LB | proxy/LB, multi-node, service discovery |
| **Zero-downtime** | Пересоздание сервиса часто с простоем | rolling / blue-green с health gates |

## Вывод

Docker Compose — отличный инструмент **описания и локального запуска** многоконтейнерного приложения. Он не заменяет оркестратор:

1. **Self-healing** — упавший инстанс не восстанавливается до желаемого числа реплик автоматически так, как в production-кластере.
2. **Scaling** — `--scale app=3` упирается в порты, отсутствие балансировщика и один хост.
3. **Zero-downtime deployment** — обновление через Compose обычно не даёт безопасного rolling-выката без простоя.

Поэтому Compose остаётся в зоне **dev / demo / простой single-host**, а для production выбирают платформы с контроллером состояния, балансировкой и стратегиями обновления (Kubernetes, Swarm и аналоги). Compose может помочь *описать* сервисы; *эксплуатировать* их под нагрузкой и отказами — уже другая задача.
