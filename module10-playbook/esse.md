# Эссе: Как Ansible Roles позволяют переиспользовать код

**Задание:** объяснить, как структура `roles/` (например, `roles/nginx/tasks/main.yml`, `roles/postgres/tasks/main.yml`) даёт возможность переиспользовать код.

---

## Проблема «толстого» playbook

Playbook удобен, когда задач немного: поставить nginx, положить конфиг, запустить сервис. Но когда появляется второй сервис (PostgreSQL), третий (Redis), а окружений несколько (dev, staging, prod), один YAML быстро раздувается. Одинаковые блоки копируются между playbook’ами, правки приходится дублировать, и «источник правды» размазывается по файлам.

Роли решают эту проблему: выносят готовый сценарий настройки компонента в отдельный переиспользуемый пакет.

## Что такое role

**Role** — это соглашение о структуре каталогов. Ansible знает, где искать задачи, шаблоны, переменные и обработчики, если они лежат в привычных местах:

```text
roles/
├── nginx/
│   ├── tasks/main.yml      # что сделать
│   ├── handlers/main.yml   # что перезапустить при изменениях
│   ├── templates/          # Jinja2-шаблоны (nginx.conf.j2)
│   ├── files/              # статичные файлы
│   ├── vars/main.yml       # переменные роли
│   └── defaults/main.yml   # значения по умолчанию
└── postgres/
    ├── tasks/main.yml
    ├── handlers/main.yml
    ├── templates/
    └── defaults/main.yml
```

Логика nginx живёт в `roles/nginx/`, логика PostgreSQL — в `roles/postgres/`. Playbook больше не описывает каждый `apt` и `template` вручную — он только **подключает** роли.

## Переиспользование: один раз написали — много раз применили

Без ролей один и тот же набор задач пришлось бы копировать в каждый playbook. С ролями достаточно:

```yaml
- name: Configure web and database
  hosts: app
  become: true
  roles:
    - nginx
    - postgres
```

Или точечно на разных группах хостов:

```yaml
- hosts: web
  roles:
    - nginx

- hosts: db
  roles:
    - postgres
```

`roles/nginx/tasks/main.yml` остаётся одним файлом. Его меняют один раз — и все playbook’и, которые подключают роль `nginx`, получают обновление. То же для `roles/postgres/tasks/main.yml`: установка, конфиг, сервис — в одном месте, а не в пяти копиях.

Так роли дают **переиспользование кода**: роль — это модуль инфраструктуры, а playbook — сценарий сборки из модулей.

## Разделение ответственности

Роли помогают не только копировать меньше YAML, но и думать о системе как о наборе сервисов:

| Роль | Отвечает за |
|------|-------------|
| `nginx` | веб-сервер, порт, конфиг, restart через handlers |
| `postgres` | СУБД, пользователи БД, listen-адрес, данные |

Playbook отвечает на вопрос «**какие** компоненты нужны этому окружению». Роль отвечает на вопрос «**как** привести компонент к нужному состоянию». Это ближе к реальной эксплуатации: nginx и postgres живут своей жизнью и настраиваются независимо.

## Параметризация вместо копипаста

Переиспользование усиливают переменные. В `defaults/main.yml` роли задают разумные значения:

```yaml
# roles/nginx/defaults/main.yml
nginx_port: 80
```

```yaml
# roles/postgres/defaults/main.yml
postgres_port: 5432
```

В конкретном playbook или inventory их можно переопределить (`nginx_port: 8080` для staging), не трогая tasks. Один и тот же `tasks/main.yml` работает в разных окружениях — меняются только входные параметры, а не код роли.

## Связь с тем, что уже есть в playbook

То, что сейчас лежит «плоско» в `nginx-playbook.yml` (apt → template → service → handlers), в роли переезжает в `roles/nginx/tasks/main.yml` и `roles/nginx/handlers/main.yml`, а шаблон — в `roles/nginx/templates/`. Поведение то же, но артефакт становится переносимым: роль можно подключить в другой playbook, отдать коллеге или выложить в отдельный репозиторий ролей.

## Вывод

Структура `roles/nginx/...` и `roles/postgres/...` позволяет:

1. **Не дублировать** задачи установки и настройки между playbook’ами.
2. **Подключать** готовые компоненты одной строкой в `roles:`.
3. **Менять логику в одном месте** (`tasks/main.yml` роли) и получать эффект везде, где роль используется.
4. **Настраивать поведение через переменные**, сохраняя один и тот же код задач.

Итог: playbook описывает *состав* системы, роли инкапсулируют *как настроить каждый сервис* — и именно поэтому код автоматизации становится переиспользуемым.
